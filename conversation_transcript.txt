USER:
On an android phone, how easy would it be to access the google timeline data for processing? What steps would need to happen for a 3rd party app to achieve this?

ASSISTANT:
Short answer: there's no clean programmatic path anymore. It went from "moderately annoying" to "manual export only" when Google changed how Timeline works.

Why it's hard now

Until late 2024, Timeline data lived in your Google Account and could be pulled via Google Takeout. That's no longer true -- the Timeline is no longer part of Google Takeout because it's no longer stored on Google's servers; the cloud backup is encrypted and you don't hold the key. Location history now lives locally on the device, and Google doesn't expose a public API for reading it.

How the data actually gets out today

The only supported route is a manual, on-device export triggered by the user: in the phone's settings under Location -> Location Settings -> Timeline, there's an "Export Timeline data" entry, which requires confirming the action and authenticating with the device's lock method, then choosing a save location. This produces a JSON file (Records.json / Semantic Location History format).

What a 3rd-party app would need to do

1. No silent access. There's no intent, content provider, or API a third-party app can call to fetch Timeline data directly -- Android's scoped storage and Google's lack of a public endpoint rule that out.
2. User must export manually first. The app has to instruct the user to go through Maps/Android Settings -> Location -> Timeline -> Export, save the JSON, then share/upload that file into the app (via Storage Access Framework file picker, share sheet, or manual upload).
3. Parse the JSON. Once you have the file, it's just structured data -- the export contains formats like Records.json, Semantic Location History, and a phone takeout format, with segments carrying confidence levels, activity-type probabilities, and waypoint paths. Existing tools like Dawarich or Timeline converters already do this parsing for reference.
4. Handle scale/perf. These files can get large -- a Records.json with 600k+ points can take 20-30 seconds to process even client-side in a browser, so a mobile app should parse asynchronously/in background.
5. No automatic refresh. Since this is a one-time manual export, there's no "sync" -- a legitimate app would need to prompt users to re-export periodically if they want updated data, or pivot to collecting fresh location data itself going forward rather than relying on Google's export.

Bottom line: for a 3rd-party app, this is a "guide the user through export -> import the file -> parse JSON" flow, not an API integration. Anything claiming to pull Timeline data automatically/in the background without that manual export step either predates Google's 2024 change or isn't accessing real Timeline data through legitimate means.

Say that I have exported the json files onto my linux desktop. How could I use this data to render jpg images that would show the route of your day on a map image that is sized to the area you were for that day. What 3rd party service would you recommend, say something like openmaps?

ASSISTANT:
For this, my recommendation is contextily (Python) backed by OpenStreetMap tiles -- it's a well-maintained wrapper specifically built for exactly this "plot a route/points on a real map background and save as an image" use case, and it works entirely on your Linux desktop with no account/API key needed for the default OSM tiles.

Why this over alternatives:
- contextily + matplotlib -- free, no API key, outputs standard image formats, auto-fits a bounding box, lets you control DPI/size precisely. Good default choice.
- Mapbox Static Images API / MapTiler / Stadia Maps -- nicer-looking tile styles, but require an API key and have usage limits/costs. Worth it only if you want prettier cartography.
- staticmaps / go-staticmaps / py-staticmaps -- lighter-weight alternatives if you don't want the geopandas/matplotlib stack, but less flexible for styling the route line.
- Note: OSM's own tile servers have a usage policy (2 max parallel connections, no heavy automated bulk use) -- fine for rendering your own days one at a time, just don't hammer it in a loop for years of data without caching or a rate limit.

Here's a working script. It handles the two common Timeline export shapes (older Records.json style with a flat list of location points, and the newer semantic/segments format with activitySegment/visit blocks) -- you may need to tweak the field names slightly once you look at your actual file, since Google has changed this schema a few times.

--- render_timeline_day.py ---

#!/usr/bin/env python3
"""
render_timeline_day.py

Reads a Google Timeline export JSON, extracts the coordinates for a given
day, and renders them as a route line on top of an OpenStreetMap basemap,
cropped/sized to the bounding box of that day's travel. Saves as a JPG.

Install deps first:
    pip install contextily geopandas shapely matplotlib pandas --break-system-packages

Usage:
    python render_timeline_day.py path/to/timeline_export.json 2026-06-14 output.jpg
"""

import sys
import json
from datetime import datetime, timezone

import pandas as pd
import geopandas as gpd
from shapely.geometry import Point, LineString
import matplotlib.pyplot as plt
import contextily as cx


def load_points_for_day(json_path: str, target_date: str) -> pd.DataFrame:
    """
    Extract (timestamp, lat, lon) rows for the given YYYY-MM-DD date.

    Handles a few known Google Timeline export shapes:
      1. Old "Records.json" style: {"locations": [{"timestampMs" / "timestamp",
         "latitudeE7", "longitudeE7"}, ...]}
      2. Semantic Location History style: {"timelineObjects": [
         {"activitySegment": {"startLocation": {...}, "endLocation": {...},
           "waypointPath": {"waypoints": [{"latE7","lngE7"}, ...]}, "duration": {...}}},
         {"placeVisit": {"location": {...}, "duration": {...}}}
         ]}
      3. Newer on-device export style with "semanticSegments" and "rawSignals"/
         "timelinePath" containing "point" as "lat,lng" strings.

    If your file's structure doesn't match, open it and adjust the field
    names below -- Google has changed this schema more than once.
    """
    with open(json_path, "r", encoding="utf-8") as f:
        data = json.load(f)

    rows = []
    target = datetime.strptime(target_date, "%Y-%m-%d").date()

    def in_target_day(dt: datetime) -> bool:
        return dt.astimezone(timezone.utc).date() == target

    # --- Shape 1: flat "locations" list ---
    if "locations" in data:
        for loc in data["locations"]:
            ts = loc.get("timestamp") or loc.get("timestampMs")
            if ts is None:
                continue
            if isinstance(ts, str):
                dt = pd.to_datetime(ts, utc=True, errors="coerce")
                if pd.isna(dt):
                    continue
                dt = dt.to_pydatetime()
            else:
                dt = datetime.fromtimestamp(int(ts) / 1000, tz=timezone.utc)

            if not in_target_day(dt):
                continue

            lat = loc.get("latitudeE7")
            lon = loc.get("longitudeE7")
            if lat is not None and lon is not None:
                rows.append((dt, lat / 1e7, lon / 1e7))

    # --- Shape 2: "timelineObjects" (Semantic Location History) ---
    for obj in data.get("timelineObjects", []):
        seg = obj.get("activitySegment") or obj.get("placeVisit")
        if not seg:
            continue
        duration = seg.get("duration", {})
        start_str = duration.get("startTimestamp") or duration.get("startTimestampMs")
        if start_str is None:
            continue
        dt = pd.to_datetime(start_str, utc=True, errors="coerce")
        if pd.isna(dt):
            continue
        dt = dt.to_pydatetime()
        if not in_target_day(dt):
            continue

        # Waypoints along an activity segment (the actual route)
        waypoints = seg.get("waypointPath", {}).get("waypoints", [])
        for wp in waypoints:
            lat = wp.get("latE7")
            lon = wp.get("lngE7")
            if lat is not None and lon is not None:
                rows.append((dt, lat / 1e7, lon / 1e7))

        # Start/end locations (also useful, especially for placeVisit)
        for key in ("startLocation", "endLocation", "location"):
            loc = seg.get(key)
            if loc and "latitudeE7" in loc and "longitudeE7" in loc:
                rows.append((dt, loc["latitudeE7"] / 1e7, loc["longitudeE7"] / 1e7))

    # --- Shape 3: newer "semanticSegments" with "point": "lat,lng" strings ---
    for seg in data.get("semanticSegments", []):
        start_str = seg.get("startTime")
        if not start_str:
            continue
        dt = pd.to_datetime(start_str, utc=True, errors="coerce")
        if pd.isna(dt):
            continue
        dt = dt.to_pydatetime()
        if not in_target_day(dt):
            continue

        path = seg.get("timelinePath", []) or seg.get("waypointPath", {}).get("waypoints", [])
        for wp in path:
            pt = wp.get("point")
            if isinstance(pt, str) and "," in pt:
                lat_s, lon_s = pt.split(",")
                rows.append((dt, float(lat_s), float(lon_s)))

    if not rows:
        raise ValueError(
            f"No points found for {target_date}. Open the JSON and check the "
            "top-level keys / field names against the parsing logic above."
        )

    df = pd.DataFrame(rows, columns=["timestamp", "lat", "lon"]).sort_values("timestamp")
    return df


def render_day(df: pd.DataFrame, out_path: str, pad_ratio: float = 0.15, dpi: int = 200):
    """Render the day's points/route on an OSM basemap, cropped to the day's bounding box."""
    geometry = [Point(xy) for xy in zip(df["lon"], df["lat"])]
    gdf_points = gpd.GeoDataFrame(df, geometry=geometry, crs="EPSG:4326").to_crs(epsg=3857)

    line = None
    if len(df) > 1:
        line_geom = LineString(zip(df["lon"], df["lat"]))
        gdf_line = gpd.GeoDataFrame(geometry=[line_geom], crs="EPSG:4326").to_crs(epsg=3857)
        line = gdf_line

    # Bounding box with padding so the route isn't jammed against the edges
    minx, miny, maxx, maxy = gdf_points.total_bounds
    dx = (maxx - minx) or 500  # fallback padding in meters if all points coincide
    dy = (maxy - miny) or 500
    pad_x = dx * pad_ratio
    pad_y = dy * pad_ratio
    minx, maxx = minx - pad_x, maxx + pad_x
    miny, maxy = miny - pad_y, maxy + pad_y

    # Keep aspect ratio sane for a JPG (roughly matches width:height to bbox shape)
    width_m = maxx - minx
    height_m = maxy - miny
    aspect = width_m / height_m if height_m else 1
    fig_h = 8
    fig_w = max(4, min(16, fig_h * aspect))

    fig, ax = plt.subplots(figsize=(fig_w, fig_h))
    ax.set_xlim(minx, maxx)
    ax.set_ylim(miny, maxy)

    if line is not None:
        line.plot(ax=ax, color="#1a73e8", linewidth=3, alpha=0.9, zorder=2)
    gdf_points.plot(ax=ax, color="#d93025", markersize=8, zorder=3)

    cx.add_basemap(ax, source=cx.providers.OpenStreetMap.Mapnik, zoom="auto")

    ax.set_axis_off()
    plt.tight_layout(pad=0)
    fig.savefig(out_path, dpi=dpi, format="jpg", bbox_inches="tight", pad_inches=0)
    plt.close(fig)


if __name__ == "__main__":
    if len(sys.argv) != 4:
        print("Usage: python render_timeline_day.py <timeline.json> <YYYY-MM-DD> <output.jpg>")
        sys.exit(1)

    json_path, target_date, out_path = sys.argv[1], sys.argv[2], sys.argv[3]
    day_df = load_points_for_day(json_path, target_date)
    print(f"Found {len(day_df)} points for {target_date}")
    render_day(day_df, out_path)
    print(f"Saved map to {out_path}")

--- end of render_timeline_day.py ---

Setup:
pip install contextily geopandas shapely matplotlib pandas --break-system-packages

Run:
python render_timeline_day.py timeline_export.json 2026-06-14 june14.jpg

What it does:
1. Parses your JSON, trying the three field-layouts Google has used for exports (flat locations, timelineObjects with activitySegment/placeVisit, and the newer semanticSegments).
2. Filters points down to just the target date.
3. Computes a bounding box around that day's points with ~15% padding so the route isn't crammed against the image edges.
4. Draws the route as a blue line with red dots for individual points.
5. Pulls OSM tiles via contextily as the basemap, sized/cropped to that bounding box, and saves the result as a JPG.

One thing to check first: open your actual JSON file and look at the top-level keys -- Google's Timeline export schema has shifted a couple of times (especially since the late-2024 move to on-device storage), so the field names in the parsing function (latitudeE7 vs point: "lat,lng" strings, etc.) may need small tweaks to match what you actually have.
